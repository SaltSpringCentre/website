How This Site Works

Site Architecture

This website is a collection of static HTML pages hosted for free on GitHub Pages. There is no WordPress, no database, no server to maintain. Five small Cloudflare Workers bridge the site to third-party services (Retreat Guru, Mailchimp, Acuity, Square, contact form relay).

Content is managed through external tools (Mailchimp, Google Sheets, Google Calendar, Retreat Guru, Square) plus a couple of in-repo files (events-data.js, posts/*.md, sscy-photos.json). When you update content in any of those places, the website reflects the change on the next page load.

Acuity Scheduling

Class & Treatment Booking

Iframe fallback in use

Yoga class registration and wellness treatment bookings are managed through Acuity Scheduling (Squarespace Scheduling). The booking widget is embedded on the Calendar and Wellness pages via an iframe pointing at Acuity's hosted scheduler (slug 5618a029).

Acuity Admin Dashboard Acuity Help Center

How it works

  • The iframe embed shows whichever category filter is passed in the URL (e.g. ?category=Yoga+Classes on calendar.html, ?category=Retreats on event pages).
  • Clients book and pay through Acuity's UI (Acuity is connected to Square for processing).
  • Admins manage classes, treatments, appointment types, and pricing inside Acuity.
About the API: there is a deployed sscy-acuity Cloudflare Worker that proxies the Acuity API, intended to render class / treatment lists in SSCY's own style instead of the iframe. It's blocked at the Acuity plan level — their API requires the Powerhouse plan. The iframe fallback works fine, so the upgrade is optional.

Where it appears

  • calendar.html › "Book a Yoga Class" section (Acuity iframe).
  • wellness.html › "Book a Treatment" section (Acuity iframe).
  • event.html › fallback booking link for local retreats not on Retreat Guru.
Square

Payments & Donations

Live — Sandbox

Donations and shop purchases are processed on-site using Square's Web Payments SDK. The card form is embedded directly in the page (PCI-safe, tokenized in-browser). A Cloudflare Worker named sscy-square bridges the site to Square's Orders and Payments APIs. Customers never leave saltspringcentre.com.

Square Dashboard Square Developer Portal

Current status

  • Checkout + donate flow live in Sandbox mode (fake cards only).
  • Tested end-to-end: add to cart → checkout → card form → receipt.
  • Donate modal has tier buttons ($25 / $50 / $108 / $200) plus a custom amount field.
  • Production flip pending: swap 4 secrets on the worker (SQUARE_ACCESS_TOKEN, SQUARE_LOCATION_ID, SQUARE_APP_ID, SQUARE_ENV) from sandbox to production values. No code change needed.
  • Monthly Gift button still uses the legacy Square Checkout Link (subscriptions need separate Square Subscriptions API integration).

Add or change a product

  1. Open shop.html in the repo. Find the PRODUCTS array in the script block.
  2. Edit the product entry (title, price, category, img, desc). Prices are in dollars with decimals (e.g. price: 25.00).
  3. Commit and push. Site rebuilds automatically.

Fees

  • 2.9% + $0.30 per online transaction (CAD, Square free plan).
  • No monthly fee.
  • All transactions visible in Square Dashboard alongside in-person sales.
Mailchimp

Newsletter & Subscribe Form

Live

Every page's footer has a "Join Our Community" email field. When a visitor submits, the sscy-mailchimp-sync Cloudflare Worker's /subscribe endpoint validates the email and adds it to the SSCY audience with pending status (double opt-in). Mailchimp sends the confirmation email; subscriber only counts once they click confirm. If the email is already subscribed, the form shows a friendly "already subscribed" message.

Mailchimp Admin

Sending a newsletter

  1. Design and send the campaign in Mailchimp as usual.
  2. The Mailchimp webhook fires on Send and triggers the sscy-mailchimp-sync worker.
  3. Worker fetches the campaign HTML, strips Mailchimp chrome, converts to markdown, commits posts/YYYY-MM-DD-slug.md + updates posts/index.json.
  4. Post appears on blog.html within ~1 minute of the site rebuilding.

Required audience settings

  • Only Email should be marked required on the audience. If other fields (Name, Address, etc.) are marked required, the API will reject subscriptions because the footer form only asks for email. The worker auto-fills empty defaults for any required merge fields it finds — but best practice is to keep only Email required.
Google Forms

Contact & Request Forms

Needs Deploy

The contact form (contact.html) is a real form, not a mailto: link. When a visitor submits, the fields are POSTed to a Cloudflare Worker (sscy-contact), which validates + spam-checks + hands off to a Google Apps Script that sends the email from info@saltspringcentre.com. The sender's email is set as the Reply-To so replies from Gmail go directly to the visitor.

Google Apps Script

Current status

  • Form HTML is live on both contact.html and m/contact.html.
  • Worker code is in the repo at workers/contact.js but not deployed yet.
  • Apps Script relay code is at scripts/contact-relay.gs but not deployed yet.
  • Submissions currently fail with a network error until the worker + Apps Script are deployed.

Deploy steps (one-time)

  1. Open script.google.com signed in as info@saltspringcentre.com (so GmailApp sends as that account).
  2. Create a new Apps Script project and paste the contents of scripts/contact-relay.gs.
  3. Deploy → Web app. Execute as: Me. Who has access: Anyone. Copy the /exec URL.
  4. In Cloudflare, create a Worker named sscy-contact and paste workers/contact.js from the repo.
  5. Add a secret CONTACT_RELAY_URL with the Apps Script URL from step 3.
  6. Done. Contact form submissions will start going through.

Other request forms

Accident/Incident, Schedule Adjustment, Time Away, Purchase Order, and Guest Request forms referenced from the Community Dashboard still use Google Forms (one link per form in the Sheet).

Mailchimp + Markdown

Blog & News

Live

The blog is no longer WordPress. 680 posts were migrated out of Flywheel into local markdown files stored in /posts/. blog.html reads posts/index.json and renders them. New newsletters auto-archive from Mailchimp — when a campaign is sent, a Mailchimp webhook fires the sscy-mailchimp-sync worker, which fetches the campaign HTML, converts it to markdown, and commits a new post to the repo via the GitHub API. The site redeploys on push. No manual steps.

Mailchimp Posts in GitHub

Publishing a new blog post

  1. Compose the newsletter in Mailchimp as usual.
  2. Send the campaign.
  3. Within a few seconds, a new posts/YYYY-MM-DD-title.md file appears in the repo and posts/index.json is updated.
  4. GitHub Pages rebuilds (~1 min) and the post is live on blog.html.

Editing an existing post

  • Open the markdown file in posts/ in the GitHub web editor.
  • Change the frontmatter (title, date, categories) or body text.
  • Commit. Site updates on rebuild.

Image assets

Blog images live in /posts/images/ — ~2,500 files compressed to 1200px wide. Original high-res versions are archived in Google Drive at My Drive/SSCY/Blog/images-original/.

Photo Picker

Image Library & Banner Pool

Live

All site images (site photos + 2,500 blog images) are catalogued in sscy-photos.json. You manage this library through photos.html: a dark admin-only page with categories, drag-and-drop, inline captions, right-click (or Ctrl+click) menus, and a search bar.

Open Photo Picker

Sort a photo into a category

Drag it. Or right-click (Ctrl+click on Mac trackpad) → pick a category from the menu.

Flag a photo for the homepage banner

Right-click a photo → "Use as banner" → pick Any event, Concerts only, or Retreats only. A gold / teal / terracotta badge appears on the thumbnail. The homepage banner picks deterministically from the flagged pool matching the next upcoming event's type.

Publish your changes

  1. Click Export JSON. Browser downloads sscy-photos.json.
  2. Replace the existing sscy-photos.json at the root of the repo with your downloaded file.
  3. Commit + push. Banner and gallery pick up the new data on next page load.

Notes

  • Your edits auto-save to browser localStorage as you work. They don't go live until you export + commit the JSON.
  • The picker merges the published JSON on load, so you'll always see the latest published library plus your local edits.
  • The 2,499 "Blog Library" items are collapsed by default — expand or use the search bar to browse.
